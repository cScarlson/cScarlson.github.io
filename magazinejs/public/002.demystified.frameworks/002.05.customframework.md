
A Pattern For Scalable Custom Frameworks
================================================================

## Myth
_You need a frontend development framework for every UI you build_.

## Reason
Both Nicholas Zakas and Addy Osmani have come up with distinct approaches for defining, in their words, "Scalable Large Scale JavaScript Architectures". Whether you use a hybrid strategy between the two or not or put your own twist on any of it, there are a few aspects to each that are worth looking into. This goes for small, boutique sites and large ESAs alike. If you forego writing your own Custom Elements, you can still start out with your own "poor man's framework" and scale it into something like what Zakas and Osmani have put forth.

Somewhat early on in my own career, I led a team of frontend developers at Oppenheimer Funds on a quest to clean up our frontend architecture for their enterprise scale marketing site. Architecture always exists but we had no _defined_ architecture. After gathering plenty of metrics for various static analysis, boomerang bugs and other things like coupling -- even inventing our own analysis model, "CAMP" (Average Coupling Per Module) -- I brought the issues to the frontend team's attention and from there the backend team to help us decide which direction to go. It was between my own flavor of a hybrid Zakas/Osmani architecture mixed with a more pure Zakas architecture. "Gen", my colleague, had Nicholas Zakas consult her team a few years before at another company, so we posed this new solution and put it up to a vote against React with the entire team of around a dozen engineers.

Our solution won and even 5 years after leaving the company I could check the console of that site for our namespace and they were still using it. That's some pretty good mileage and they may only have ditched it because it wasn't hip enough. That was not the first time I had written scalabl custom frameworks for clients & companies who were very satisfied with it. A long time later, I came up with an even simpler pattern using `HTMLUnknownElement`s.

## A Poor man's custom component framework

### Basic
With just a couple of `function`s, we can have liftoff.

```typescript
class App {
    static tagName = 'app';
    static content = `<div>app<child></child></div>`;
}
```

```typescript
class Child {
    static tagName = 'child';
    static content = `<div>child</div>`;
}
```

```typescript
function bootstrap(Module) {
    const { tagName, content } = Module;
    const elements = this.getElementsByTagName(tagName);
    const instances = [ ...elements ].reduce(mount.bind(Module), []);
    
    return instances;
}
    
function mount(instances, element) {
    const { content } = this;
    const instance = new this(element);
    const collection = [ ...instances, instance ];
    
    element.innerHTML = content;
    collection.push( ...modules.map( bootstrap.bind(element) ) );
    
    return collection;
}
```

```typescript
const instances = [ App, Child ].map( bootstrap.bind(document) );
```

For its size, this _microframework_, if you will, packs plenty of punch -- especially, if all you need it for is a small boutique site that'll never scale beyond 5 pages and, if it does, it's so simple and terse that nobody will have a problem translating it to another framework. We could also add some basic bells & whistles here that would make it much more pleasurable to work with -- our own interpolation or something like Handlebars, automatic event-handler bindings, a `MutationObserver`, or types of property bindings. In the next section, we wrap the injected element in a _Sandbox_ to give us a bit more lift.


### Using A _Sandbox_ (Nicholas Zakas)

#### Framework Core: `@core/core.ts`
```typescript
type Options = { target: HTMLElement } & any;

export class Core {  // Core IS the Sandbox
    static $components: Map<string, any> = new Map();
    private Component: any = Object;
    private tagName: string = '';
    public target: HTMLElement = document.createElement('div');  // use default div
    get dataset() { return this.target.dataset }
    get classList() { return this.target.classList }
    
    constructor(options: Options = {}) {
        var options = { ...this, ...options };
        const { Component, tagName, target } = options;
        
        this.Component = Component;
        this.tagName = tagName;
        this.target = target;
        
        return this;
    }
    
    static bootstrap(Component: any) {
        const { tagName } = Component;
        const $targets = document.getElementsByTagName(tagName);
        const targets = Array.prototype.slice.call($targets);
        const instances = this.mount(Component, ...targets);
        
        return this;
    }
    
    static mount(Component: any, target?: HTMLElement, ...more: HTMLElement[]) { // Tail-Call Optimized
        if (!target) return Core.$components.get(Component.tagName);  // early-return
        const { $components } = Core;
        const { tagName } = Component;
        const $ = new Core({ Component, tagName, target });  // sandbox instance
        const instance = new Component($);
        
        if ( !$components.has(tagName) ) $components.set(tagName, [ ]);
        $components.get(tagName).push(instance);
        
        if (more.length) return this.mount(Component, ...more);
        return $components.get(tagName);
    }
    
    on(type: string, handler: EventListenerOrEventListenerObject, capture: boolean = true) {
        const { target } = this;
        target.addEventListener(type, handler, capture);
        return this;
    }
    
    off(type: string, handler: EventListenerOrEventListenerObject, capture: boolean = true) {
        const { target } = this;
        target.removeEventListener(type, handler, capture);
        return this;
    }
    
    fire(type: string, data?: any) {  // dispatches on host component
        const { target } = this;
        const e = new CustomEvent(type, { detail: data });
        
        target.dispatchEvent(e);
        return this;
    }
    
    load(content: string) {  // loads innerHTML from string
        const { target } = this;
        target.innerHTML = content;
        return this;
    }
    
    style(styles: string) {  // injects styles in document head if not already existing
        if ( document.head.querySelector(`#${this.tagName}`) ) return this;
        const { tagName } = this;
        const { head } = document;
        const style = document.createElement('style');
        
        style.id = tagName;
        style.innerHTML = styles;
        head.appendChild(style);  // or append it inside of this.target
        
        return this;
    }
    
};
```

Notice that the static `mount` method takes advantage of Nicholas Zakas's "Sandbox" pattern by passing an instance of `Core` (`$`) into the component being bootstrapped, which _is_ the module's Sandbox. We could have added more sandbox utilities to the sandbox, such as an `interpolate` method, but this should suffice for bare minimum. In fact, we could even have the `load` method assume the use of a framework like Handlbars if we want. Here's what the usage might look like.

#### Usage: Registration
##### `./index.ts`
```typescript
import { Core } from './src/app/core';
import { App } from './src/app/app.component';
import { Child1 } from './src/app/subsystem/child1';
import { Child2 } from './src/app/subsystem/child2';

function bootstrap() {
    Core.bootstrap(App)
        .bootstrap(Child1)
        .bootstrap(Child2)
        ;
}

window.addEventListener('load', bootstrap, false);
if (document.readyState === 'complete') bootstrap();
```


#### Usage: HTML

##### `./index.html`
```html
<!doctype html>
<html lang="en">
    <head>...</head>
    <body>
        <app></app>
    </body>
</html>
```

##### `@app/app.component.ts`
```typescript
import { Core } from './core/Core';
import styles from './app.component.css';
import content from './app.component.html';

class App {
    static tagName: string = 'app';
    
    constructor(private $: Core) {
        $.style(styles).load(content);
    }
    
}

export { App };
```
##### `@app/app.component.html`
```html
<div class="app root">
    <h1 class="app header">Sample App</h1>
    <main>
        <child1></child1>
    </main>
</div>
```

We basically just define a static member, "`tagName`", on our component definitions so that the Core can look them up in the DOM using `document.getElementsByTagName`. We add some convenience methods onto the core, although, we could have the engine take care of initial content loading automatically by defining a static "`content`" property on our component definitions (like `tagName`). As you can see, this [very] small system already grants us a lot of leverage.


### Using An Event-Driven Mediator (Addy Osmani)
If we want to incorporate a mediator (see _The Mediator Pattern_) for a cross-component Event-Bus, it is rather easy.

#### Mediator: `@core/utilities/patterns/behavioral/mediator.ts`
```typescript
const target = new EventTarget();  // use singleton instance

export class Mediator {
    
    dispatch(channel: string, data: = {}): this {
        const event = new MessageEvent(channel, { data });
        target.dispatchEvent(event);
        return this;
    }
    
    publish(channel: string, data?: any): this {
        if (channel in this) return this[channel](channel, data);
        return this.dispatch(channel, data);
    }
    
    subscribe(channel: string, handle: EventListenerOrEventListenerObject): this {
        target.addEventListener(channel, handle, true);
        return this;
    }
    
    unsubscribe(channel: string, handle: EventListenerOrEventListenerObject): this {
        target.removeEventListener(channel, handle, true);
        return this;
    }
    
};
```

#### Framework Core: `@core/core.ts`
```typescript
import { Mediator } from '@core/utilities/patterns/behavioral';

type Options = { target: HTMLElement } & any;

export class Core extends Mediator {  // Core IS the Sandbox
    ...
};
```

## Synopsis
We demonstrated that there are scalable patterns for large scale frontend architecture and provided an example of what one flavor might look like (for inspiration). Remember, all these other frameworks started with someone else experimenting with their own patterns and then iterating on the ones that worked best. We are _not_ suggesting to reinvent the wheel at your company's expense but, when you go to build out that boutique site pro bono for your sister in law's solan, pay yourself off a little with experience and you might just invent the next best frontend development framework, SDK, or other patterns that will take your career to the next level. When you do that, you'll realize that oftentimes you don't really need a typical frontend development framework, you just need something that does what you need, including scaling.
