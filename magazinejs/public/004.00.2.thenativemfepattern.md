
The Microfrontend Pattern &copy;
================================================================


### Intent
Leverage `iframes` as self-aware services that externalize themselves as UI components in place of the original iframe element.

### Also Know As
- The MFE Pattern
- The Native MFE Pattern
- The Partials Pattern
- The Includes Pattern
- Remote Module Definition (RMD)
- Remote Component Definition (RCD)
- Remote Element Definition (RED)
- Web As Plugin Architecture (WAPA)

### Motivation
Cumbersome Microfrontend frameworks like RSBuild and Module Federation provide MFE functionality and scalability to large or complex websites and web applications alongside the tooling they provide.

But there's a problem with this approach: frameworks like these thwart their own scalability by requiring extra infrastructure, additional learning & developer onboarding, build & packaging utilities, and different plugins for each development framework, some of which may be custom for a particular use-case which causes more ifrastructure and learning curve. The Microfrontend Pattern describes how to leverage sandboxed Browsing Contexts so that clients written in virtually any framework (or native) can implement microfrontend architecture components & UI services across The Web.

### Applicability
Use the MFE pattern when
- you want to fractionate your website or webapp into disparate components.
- you want to separate your components into domains owned by different teams.
- you want sandboxed components that initialize in separate Browsing Contexts.
- you want to share components across multiple websites and webapps.
- you want to publish components & UI services for others across The Web to use.
- you want to consume components & UI services from others across The Web for your own use.

### Structure
```
     Site A
┌────────────────┐                           Site B                                    Site C
|  ⧉ App Shell  |                         ┌────────┐                                ┌────────┐
│       ⊡────────────────────────────────▸|  RMD ⊡────────────────────────────────▸|   RMD  |
|       |        |                         └────────┘                                └────────┘
└───────|────────┘
        |
        |
        ▾
    ┌────────┐
    │   RMD  │ Site A
    └────────┘
```

### Participants
- App Shell (Root Branch Node)
  - Defines its own MFEs (RMDs).
  - Consumes its own MFEs (RMDs).
  - Consumes other MFEs (RMDs).
- RMD (Branch or Leaf Node).
  - Defines a Document Fragment which an App Shell expects.
  - Replaces its `frameElement` with the Document Fragment.

### Collaborations
- The App Shell is a website or webapp that acts as the main host of RMDs. It is the Root Node and defines its own RMDs it wants to consume in addition to the option to consume RMDs from other sites.
- An RMD expects to be consumed by an App Shell or another RMD as a Document Fragment or some other type of UI service.

### Consequences
The Microfrontend Pattern
- Defines Document Fragment hierarchies of components & UI services into a composite, root node.
- Facilitates Lazy-Loading of RMD components & UI services.
- RMDs replace the existing frame or hide or move it to run _idly_ when the Browsing Context should continue run undisposedly.
- Advances The Web by connecting sites & apps in a federated manner for UI nodes, **turning The Web into a _Plugin Architecture_**.
- Facilitates _Skeleton UIs_ while components are loading.

### Implementation
There are specific tradeoffs to consider when electing to implement The Microfrontend Pattern.
1. _Partials & Includes_. Partials can simply select a DOM `Node` to serve the App Shell.
  - `Element`: e.g: `frameElement.replace( document.querySelector('{selector}') )`.
  - `Text`: a specific `Text` `Node` (or string) can be used to replace the frame.
  - `Attr`: an `Attr` `Node` can be attached to the `frameElement` to have a `MessageEvent` dispatched on it.
  - `Comment`: a `Comment` `Node` can be used to signal to a `MutationObserver` or otherwise in the App Shell.
  - `HTMLStyleElement`: a `style` element can be used to repace the frame to provide SDK styles or otherwise.
1. _Components_. a Web Component or one written in a different framework can be used to replace the frame. Components can leverage anything in their own Browsing Context such as `<template>`s, `<style>`s, `<script>`s, or any other faculty available.
1. _Frame Disposal_. frames can be hidden, moved, removed or replaced depending on whether the RMD needs the Browsing Context to remain undisposed for ongoing processes associated with that Browsing Context, if only so that errors do not get swallowed during development.
1. _Work Distribution_. applications & sites can be split among different teams so that they each can work on separate domains.
1. _Plugin Architecture_. plugins can be built for internal business purposes or collaboration between an organization & its clients.
1. _Performance_. Performance can suffer depending on idle frames. If all RMDs leave an idle frame, the number of DOM Nodes doubles.
1. _Slots_. usage of `HTMLSlotElement`s is generally unavailable because all `childNodes` of an `iframe` are stored only as `innerHTML` (and `innerText`). These can be 'resuscitated' into copies of the intended nodes and passed using _manual slotting_. **However**, consumers can define the expected Custom Element (with the iframe as a child) while injecting slotted children into the Web Component as normal. Services can automatically slot the [child] `iframe` into its Shadow DOM when preferable; otherwise, the iframe's selector can match `iframe[slot="disposal"]` for opt-(in/out) functionality by the consumer.
1. _Attributes_. attributes can be defined on the iframe that the provider expects. That is, the iframe can define a `DOMTokenList` for the `className`, `dataset` attributes, or any other which the provider can use for configuration.
1. _Skeleton UIs_. _Skeleton Pages_ and other UI _Skeletons_ can be automatically implemented simply by styling the iframes based on a targeted CSS Selector.

### Sample Code

#### App Shell
```html
<div class="my view">
    <iframe src="https://cors.enabled.app/public/v0.0.1/raw.element.html"></iframe>
    <my-remote-component></my-remote-component>
</div>
```

#### RMD
```html
<body>
    <template id="template">
        <h1>Remote Module Definition</h1>
    </template>
    <style id="style">
        :host {
            color: firebrick;
        }
    </style>
    <script>
        const { parent, frameElement } = window;
        const { customElements, HTMLElement } = parent;  // ensure is from parent frame
        const { log } = console;
        const TAGNAME = 'my-remote-component';
        const defined = !!customElements.get(TAGNAME);
        
        if (!defined) customElements.define(TAGNAME, class RemoteComponent extends HTMLElement {
            
            constructor() {
                super();
                this.attachShadow({ mode: 'open' });
                this.#setup();
            }
            
            #setup() {
                const template = document.querySelector('#template');
                const style = document.querySelector('#style');
                
                this.shadowRoot.appendChild(style);
                this.shadowRoot.appendChild(template.content);
                frameElement.remove();
            }
            
            connectedCallback() {
                log(`@REMOTE COMPONENT DEFINITION -> connected`);
            }
            
        });
    </script>
</body>
```

#### Output
```html
<div class="my view">
    <my-remote-component>
        #shadow-root (open)
            <style id="style">
                :host {
                    color: firebrick;
                }
            </style>
            <h1>Remote Module Definition</h1>
    </my-remote-component>
</div>
```

### Known Uses
Perhaps to best known example of usage for The MFE Pattern is from the _Frameless_ framework. Frameless&trade; is the Microfrentend framework provided by the _AsXS_ SDK. In fact, the entire AsXS&trade; SDK leverages this pattern for all of its UI components. Learn more at the [AsXS SDK and Frameless framework](https://cscarlson.github.io/asxs).

### Related Patterns
Examples of The MFE Pattern, aside from native Web Components, are partials as described above, where the RMD simply replaces the frame with static HTML or text. Otherwise, components written in [say] _Angular_, _React_, _Vue_, or any other conceivable framework can feasibly be used to run in one Browsing Context (the child frame) and transfer themselves to their respective position (of the child frame) in the DOM of the consumer declaring the iframe.
