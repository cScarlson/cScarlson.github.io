## Interpolation
If not using a Web Components framework such as Lit, one may elect to use basic interpolation on the `<template>` and `<style>` elements' `innerHTML` before attaching them to the Custom Element's `shadowRoot`. This would allow you to leverage dynamic styling in your component, based on [say] attributes, where perhaps (if ever) native CSS Attribute Selectors were not an option.

## Versioning
With a basic CI/CD system, RCDs can be deployed to specific locations that reflect a version of the component, allowing for graceful deprecation and iterative enhancements. In the example above, the RMD is published behind version `v0.0.1`, which can be a simple S3 Bucket or a simple folder in your GitHub Pages website.


## Importance
The importance of this pattern cannot be overstated. This pattern, beside saving you from the headache of cumbersome MFE frameworks like RSBuild/RSPack + ModuleFederation, **turns The Web into a Plugin Architecture**. That is, anyone anywhere can publish and consume native Web Components that integrate with anyone else's website or webapp anywhere on The Web. It can be used for Custom Elements or simple partials. Authors can develop their own protocols & patterns, such as providing a sibling frame that loads a README for usage. However, _this pattern does come with certain security concerns that you and your team need to discuss if using RCDs from untrusted origins_.

## Variants & Patterns
- Replacing the `frameElement` with a Web Component `Node` (instead of the View declaring it).
- Using attributes from the `iframe`/`frameElement` as parameters/communication for the custom element or other configurations.
- Using `<slot>`s with the `iframe`/`frameElement`'s `innerHTML`.
- Replacing the `frameElement` with a `Node` from the `<template>`'s `content` or another element (instead of the View declaring it) for loading simple partials.
- Calling a parent-frame script to define or load the element through a framework.
- Simply providing a `<template>` and `<style>`s.
- Providing a `<meta>` element whose `name` is a `tagName` for the parent frame to define (see Frameless&trade;).
- Providing a sibling README frame for documentation & usage.
- Simply replacing the `frameElement` with a `<style>` that loads an entire CSS framework.
- Simply replacing the `frameElement` with a `<style>` that loads different parts of a CSS framework.
- Many others.

## Other Considerations
This pattern, generally, only works with native Web Components. Angular, on the one hand, does provide the ability to export components as native Web Components, which would allow teams to write in one syntax and still take advantage of this pattern. However, frameworks like React do not yet provide this capability at the time of this writing, in which case your team may but stuck with a cumbersome framework. That said, it is likely feasible that one can replace the iframe with a fully-functional React component, or likewise written in any other framework.

## Synopsis
This pattern, while the RMD itself must be a Web Component, can be used in any app using any other framework because native is compatible with anything. It is a lightweight approach to achieving microfrontend architecture without cumbersome frameworks; all you need is a CI system. From there, the sky is the limit of what you can do as the Web Component author and the consumer need only know how the author expects or enforces its consumption. It can be used for Web Components, partials, atomic buttons and other components, or Feature-Complete Microfrontend views at large. Additional patterns & conventions will develop on top of this baseline pattern -- packaging & documentation conventions, Smart vs Dumb components & partials, and likely many others. Again, most frameworks & libraries are there simply to conceal a pattern and deceive the consumer into believing there is all sorts of "magic" happening behind the scenes that is usually a very small set of simple patterns.
