
Reactive Data
================

## Myth
_You need a library like RxJS, Signals, or otherwise to have Reactive Data_.

## Reason
Data Reactivity can be accomplished in roughly 25 LOC using _The Observer Pattern_.

## The Observer Pattern
Rather than pulling in RxJS (or similar), you may just have one small area of the code that needs some reactivity. RxJS _says_ it is a new and improved version of _The Observer Pattern_. However, you might say it is more so just _inspired_ by The Observer Pattern as it has zero alignment with the actual pattern's interface. The only thing RxJS and The Observer Pattern really have in common is _reactivity_. When all you need is that, you may be better off writing a _Subject_ -- the main participant of The Observer Pattern.


### To Spec

#### `@core/utilities/patterns/behavioral/observer.ts`
```typescript
interface IObserver {
    update(state: any);
}

class Subject {
    state: any = null;
    observers: Set<IObserver> = [ ];
    
    constructor(state: any = this.state) {
        this.state = state;
    }
    
    attach(observer: IObserver) {
        this.observers.push(observer);
    }
    
    detach(observer: IObserver) {
        for (let i = this.observers.length; i--;) if (this.observers[i] === observer) this.observers.splice(i, 1);
    }
    
    notify() {
        for (let observer of this.observers) observer.update(this.state);
    }
    
}

export type { IObserver };
export { Subject };
```

That is basically it for The Observer Pattern. You have a structure that accepts "observers". Observers are just objects with an `update` method. The `Subject` calls its `notify` method whenever its `state` changes. Observers can detach themselves for cleanup and we iterate backward in the `detach` method to not modify the `observers` collection in place while we `splice`. That said, here are the deviations from the pattern that reflect how I like to write my Subject class so that anything can extend it and provide more flexibility for Observers...

### Recommended Deviations from The Spec

#### `@core/utilities/patterns/behavioral/observer.ts`
```typescript
interface IObserver {
    call(state: any);  // use `call` instead of `update`. allows us to invoke functions or objects with a `call` method.
}

class Subject {
    ['undefined']: any = null;
    observers: Set<IObserver> = new Set();  // use JS Set
    
    constructor(state: any, private __key__: string|undefined) {  // allow 'key' for targeting which variable is of interest
        this[__key__] = state;  // set whichever variable reflects subject's state
    }
    
    attach(observer: IObserver, notify: boolean = true): Subject {  // use notify flag for automatically providing current state
        const { state, observers } = this;
        
        observers.add(observer);
        if (notify) observer.call(state, state);  // notify observer as soon as attached
        
        return this;
    }
    
    detach(observer: IObserver): Subject {
        const { observers } = this;
        observers.delete(observer);
        return this;
    }
    
    notify(state: any = this[this.__key__]): Subject {  // use default with ability to override
        const { observers } = this;
        observers.forEach( observer => observer.call(state, state) );  // use `call` instead of `update`
        return this;
    }
    
}

export type { IObserver };
export { Subject };
```

#### `**/observer.class.ts`
```typescript
import { type IObserver, Subject } from '@core/utilities/patterns/behavioral/observer';
import { store } from '@core/store';

class ObserverClass implements IObserver {
    data: Subject = new Subject({}, 'state');
    
    constructor(...) {
        this.data.attach(this);
        this.data.state.x = 'test';
        this.data.notify();
    }
    
    call(state: unknown) { ... }
    
}
```

#### `**/observer.function.ts`
```typescript
import { store } from '@core/store';

function handleStateChange(state: unknown) { ... }

store.attach(handleStateChange);
```

## Synopsis
As you'll notice, we can use both `function`s _and_ `class`es as our observers because we deviated a little bit from The Observer Pattern to look for a `call` method instead of an `update` method -- which `function`s already have. We also added a `notify` flag to the `attach` method to control whether or not the observer should be automatically invoked when attached. Moreover, we made `Subject` more extensible by giving subclasses the ability to configure which _key_ should be used to store the subject's _state_ rather than being married to only using a variable called "`state`". With only about 25 LOC (lines of code) we instantly breathed _reactivity_ into our app with full control over its maintenance and evolution. I encourage you to implement "observables" in this way next time you need a little reactivity in your app. Who knows?... Maybe the evolution of _your_ Subject will become the next best-selling reactive library on GitHub.
