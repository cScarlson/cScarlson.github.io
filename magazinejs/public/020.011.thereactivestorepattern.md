
The Reactive Store Pattern &copy;
================================================================


### Intent
Define a simple system which receives dispatches of data and leverages a reducer to process each dispatch while notifying observers when its internal state has changed.

### Also Know As
- Native reactive store
- Redux-style store

### Motivation
Frameworks like Redux can be used to wire up automatic reflows of module lifecycle for React components, those for other frameworks like Angular, or even native JavaScript.

But most tools like Redux provide more than most (or all) apps really need and radically complicate the boilerplate of the modules it touches. Likewise, developers are oftentimes merely looking for a simply signaling system between state and a module.

### Applicability
Use the reactive store pattern when
- you want a simple tether between a data-structure and a module to reflect updates.
- you want to define your own processors that handle how the overall state should be applied based on the type of payload.
- you want transparency of how different actions reflect different states of a data-structure.
- you want to reduce external dependencies on 3rd party libraries for security reasons or weight.
- you want full control over the internals of the Reactive Store solution for maintenance, scalability, or other unique needs.
- you want objects or functions to both receive state-change signals in a normalized way between the two.
- you want to always cache IO operations of a data-structure (in Session Storage, etc) for Fault Tolerance or (un/re)do behavior.
- you want notifications for immutable data.

### Participants
- Core Store
  - Defines base functionality for Concrete Stores.
  - Extends a _Subject_ (see The Observer Pattern) or otherwise achieves signaling & notification capabilities.
  - Encapsulates a Concrete Reducer.
  - Implements a `dispatch` method for Clients to provide data for particular types of operations.
  - Notifies client observers, preferably only if an immutable state has changed.
- Core Reducer
  - Defines base functionality for Concrete Reducers.
  - Implements an `execute` method which maps to Reducer methods that return a new state of a Concrete Store.
- Concrete Store
  - Extends the Core Store
  - Provides overrides for a particular app or use-case when necessary.
- Concrete Reducer
  - Extends the Core Reducer
  - Provides overrides for a particular app or use-case when necessary.
  - Implements hook methods that the Core Reducer maps to for handling particular dispatches of data.
  - Overrides the Core Reducer's `execute` method for (pre/post)-operations, such as Session or Local Storage caching.
- Client (Observer)
  - Subscribes to a Concrete Store andor dispatches data to the store through its `dispatch` method.
  - Implements a `call` method which the Core Store invokes to notify the module of a state change.

### Collaborations
- The Core modules, Store & Reducer, exist to provide base functionality. These are abstract but can be extended to provide more common capabilities across applications for a particular organization.
- The Concrete modules, Store & Reducer, exist to provide specific functionality for a particular application. They can also continue to be extended to provide more isolated behavior for subdomains of that application.
- Concrete Stores can encapsulate other stores as part of their state where a specialized reducer calls `dispatch` on the child store to produce a new substate.
- Clients can register themselves to handle changes in the store's state.

### Consequences
The reactive store pattern
- Leverages The Observer Pattern, The Publish-Subscribe Pattern, or other means (at its root) of notifying modules of changes in state.
- Defines one or more abstract andor concrete layers of Store-Reducer pairs to facilitate IO of state-change operations and notification thereof.
- Facilitates transparency of data-change IO.
- Uses superclasses that can be overridden to provide solutions to more & more specific needs for subdomains.
- Uses specific overrides of The Observer Pattern to
  - Normalize functions and objects as observers.
  - Provide automatic notification of observers upon registry.
- Can leverages _immutability_ to determine whether or not observers should be notified.
- Can leverage _The Composite Pattern_ for higher-grain control over data IO operations.
- Allows Stores & Reducers to implement their methods, `dispatch` & `execute` (respectively) as _Arrow Functions_ to simply provide the minimal basic external public interface necessary for other modules to interact with in a Functional Programming style.
- Allows the Core Store & Reducer and the Concrete Store & Reducer to be one in the same if the use-case will not scale beyond a simple website or webapp with simple needs. That is, all encapsulation between the Store & Reducer can be added to one of each type, blending Core and Concrete behavior into each one's particular class.
- Allows the Concrete Store to be initialized as private encapsulation within an _Application Head_ (or Subdomain Head) by leveraging _The Mediator Pattern_ in order to implement _The Custom Event Loop Pattern_ so that state-dispatches can trigger Side Effects before the store gets updated, protecting the Application Core from direct manipulations through all other modules, creating better security for your system and making your components more portable.

### Implementation
There are specific tradeoffs to consider when electing to implement The Reactive Store Pattern.
1. _Superclasses_. Superclasses must carefully define baseline functionality, depending on their scale.
  - Immutability: if lower subdomains demand a different preference in notification based on mutability, more sublayers of superclasses may be necessary alongside particular overrides.
  - Composition: when composing superclasses at different scales, the developer must decide whether or not more superclass layers should be added, whether or not a concrete child store should be part of the encapsulation of a parent store, or if The Composite Pattern should be leveraged instead.
1. _Notifications_. Stores & Reducers can collaborate to provide advanced functionality such as only notifying observers based upon which part of the state was updated.
1. _Registry_. Clients can be registered and subscribe in different ways. The developer can decide whether to use a Subject from The Observer Pattern, an `EventTarget` (perhaps wrapped/extended by `publish`, `subscribe`, and `unsubscribe` methods), extend an `EventTarget` itself, or another means of registry & notification between the Store and Clients.

### Sample Code

#### Subject (see The Observer Pattern)
##### `@core/utilities/patterns/behavioral/observer.ts`
```typescript
interface IObserver {
    call(state: any);  // use `call` instead of `update`. allows us to invoke functions or objects with a `call` method.
}

class Subject {
    ['undefined']: any = null;
    observers: Set<IObserver> = new Set();  // use JS Set
    
    constructor(state: any, private __key__: string|undefined) {  // allow 'key' for targeting which variable is of interest
        this[__key__] = state;  // set whichever variable reflects subject's state
    }
    
    attach(observer: IObserver, notify: boolean = true): Subject {  // use notify flag for automatically providing current state
        const { state, observers } = this;
        
        observers.add(observer);
        if (notify) observer.call(state, state);  // notify observer as soon as attached
        
        return this;
    }
    
    detach(observer: IObserver): Subject {
        const { observers } = this;
        observers.delete(observer);
        return this;
    }
    
    notify(state: any = this[this.__key__]): Subject {  // use default with ability to override
        const { observers } = this;
        observers.forEach( observer => observer.call(state, state) );  // use `call` instead of `update`
        return this;
    }
    
}

export type { IObserver };
export { Subject };
```

#### Core Store
##### `@core/store/store.ts`
```typescript
import { type IObserver, Subject } from '@core/utilities/patterns/behavioral/observer';
import { type Action, Reducer } from './reducer';

export class Store<T = any> extends Subject {
    data = {};
    
    constructor(public data: T = {}, private reducer: Reducer) {
        super(data, 'data');
    }
    
    dispatch(action: Action) {
        const { reducer: { execute }, data } = this;
        
        this.data = execute(data, action);
        if (this.data !== data) this.notify();
        
        return this;
    }
    
};
```

#### Core Reducer
##### `@core/store/reducer.ts`
```typescript
export type Action = { type: string, payload: any };
export class Reducer<T = any> {
    
    execute = (state: T, action: Action) => {
        const { type } = action;
        if (type in this) return this[type](state, action);
        return state;
    };
    
};
```

#### Concrete Store
##### `@app/core/store/store.ts`
```typescript
import { Store as Core } from '@core/store/store';
import { Reducer } from './reducer';
import { State } from './state';

const INITIAL_STATE = new State({ ... });
const reducer = new Reducer();

export const { dispatch } = new (class Store extends Core {})(INITIAL_STATE, reducer);
```

#### Concrete Reducer
##### `@app/core/store/reducer.ts`
```typescript
import { type Action, Reducer as Core } from '@core/store/reducer';
import { State } from './state';
export class Reducer<T = State> extends Core<T> {
    
    execute = (state: T, action: Action) => {
        const latest = super.execute(state, action);
        const persisted = JSON.stringify(latest);
        
        localStorage.setItem('persisted', persisted);  // uses subclass as middleware
        return latest;
    };
    
    ['SOME:DATUM:CHANGED'](state: State, action: Action): State {
        const { payload: datum } = action;
        return { ...state, datum };
    }
    
};
```

#### Client (Observer)
```typescript
import { dispatch } from '@app/core/store';

export class Component extends Framework {
    handleInputChange(e: InputEvent) {
        const { target } = e;
        const { value } = target;
        dispatch({ type: 'SOME:DATUM:CHANGED', payload: value });
    }
};
```

#### Using an `EventTarget` for The Publish-Subscribe Pattern (Optional)
Instead of the Store extending a `Subject`, it can extend a modified version of an `EventTarget` to implement The Publish-Subscribe Pattern.

```typescript
export class PublishSubscribe extends EventTarget {
    
    publish(channel: string, data?: any): this {
        const event = new MessageEvent(channel, { data });
        this.dispatchEvent(event);
        return this;
    }
    
    subscribe(channel: string, handle: EventListenerOrEventListenerObject): this {
        this.addEventListener(channel, handle, true);
        return this;
    }
    
    unsubscribe(channel: string, handle: EventListenerOrEventListenerObject): this {
        this.removeEventListener(channel, handle, true);
        return this;
    }
    
};
```

### Known Uses
The [AsXS &trade; SDK](https://cscarlson.github.io/asxs) provides the baseline Core Store and Core Reducer out of the box. Other versions, whether to-spec or not, are ultimately rolled up behind frameworks like Redux and similar libraries & packages.

### Related Patterns
Other similar patterns can be found in MV* frameworks such as KnockoutJS or more recent frameworks that provide basic state-to-client signaling, not limited to RxJS. However, this pattern as it's presented radically minimizes the overhead and dependency between such frameworks. There are virtually no other patterns that achieve this behavior for State Management directly; only patterns like The Observer Pattern, The Publish-Subscribe Pattern, and similar which are used as _part_ of this pattern are know at this time.
