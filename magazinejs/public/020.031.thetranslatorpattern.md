
The Translator Pattern &copy;
================================================================


### Intent
Define a Translator service that expects flat JSON files for locales and leverages The Chain of Responsibility Pattern so specific texts can be targeted by specialized translators. Translators can target LTTs (Large Text Translations) such as Markdown files for TAC (Terms And Conditions), basic string interpolation, Taxon Substitution, and any other custom use-cases.

### Also Know As
- The Internationalization Pattern
- The Localization Pattern
- The I18n Pattern
- The L10n Pattern

### Motivation
Many i18n/l10n libraries & framework plugins, such as FormatJS, provide many features out of the box such as _string interpolation_ and _nested locale structures_.

But these APIs tend to short-circuit on their scalability. They tend to complicate their APIs & codebases by providing so much that they have to predict every consumer's needs and force uncomfortable APIs and conventions on the consumer. There is a better way as long as we sacrifice one luxury: _nesting_.

### Applicability
Use the translator pattern when
- you want to minimize 3rd party dependencies.
- you want to finer control over your own processors for specific locale texts.
- you want to use your own string interpolation syntax.
- you need to conduct Large Text Translations for entire text files, such as Terms And Conditions text provided by Legal teams.
- you need to parse Markdown contained in a translation text.
- you want to define taxa for substitutions within the same locale file.
- you want to translate a large Markdown file that uses both data and static taxon interpolations based on a proprietary syntax.

### Participants
- TranslationService (API): Defines methods for translating text and changing languages.
  - Defines a `translate` method that takes a Translation Key and returns a Translation Value.
    - The `translate` method also takes an optional _scope_ parameter.
  - Defines a `change` method that takes a Language Code or Dialect Code and maps it to a Locale object.
  - Implements _The Chain of Responsibility Pattern_ to run translations through Translators from the Registry.
- Translator (middleware plugins): Functions or Objects with a `call` method that return a Translation Value.
  - Accepts a default Translation Value and an _options_ object that contains the current dialect, locale, Translation Key, and an optional scope object that reflects any additional data necessary for the Translator or the Translation Value.
- Registry
  - A `Set` object that houses all active Translators.
- Locale (Translation Map): A flat JSON file containing all translations of a given language for a given domain.
  - Defines fully-qualified Translation Keys.
  - Stores Translation Values reflected by each Translation Key.
  - Can store _Taxa_ Translations used for substitution of targeted _taxon_ that exists in other Translation Values.
    - Conventionally having the Translation Key prefixed with `TAXON:`.
  - Conventionally, named as `locale.{lang}.json` where `{lang}` is a Language or Dialect Code (`en`, `en-US`, `es`, etc).

### Collaborations
- The TranslationService receives a configuration object that reflects all initial & default values to begin translating text including all locales for target languages.
- Invoking the `translate` method of the TranslationService simply maps to the immediate value of the current locale. Therefrom, the current Translation Value, alongside a 'details' object, is passed along the chain of Translators from the Registry (see The Chain of Responsibility Pattern). Translators can decide whether or not they are interested in the operation based on the translation `key` provided in the details object. Translators can also determine interest based on data from the `scope` provided in the details object.
- Translators can be _chained_ together to perform more complex and sequential operations to fulfill the desired result of a translation target.
- Invoking the `change` method of the TranslationService simply maps a language or dialect code to a specific locale object in order to switch to different languages. For example, a UI Toggle element may be used to switch from English to Spanish, at which point the toggle would call this method, passing in `en`, `en-US`, `es`, or `es-MX` (depending on your setup & needs).

### Consequences
The Translator Pattern
- Provides human readable text for specific languages, based on a Translation Key that exists across all locale files.
- Implements The Chain of Responsibility Pattern (The CoR Pattern) under the hood to give specialized Translators a chance to modify the final output of calling the `translate` method.
- Can be modified to handle asynchronous operations involved in a translation.
- Can scale to perform any localization need by providing more Translators.
- Should be implemented with baseline, common Translators for string interpolation from the `scope` object, injection of the locale object ,itself, for Taxa, and the ability to translate large texts that exist in files which likely contain markdown.
- Developers must derive their own standards as to how much logic around taxon substitution should occur within the locale file as opposed to writing that logic in the modules invoking the TranslationService.

### Implementation
The Translator Pattern should necessarily be implemented alongside several conventions and considerations.
1. _Flat JSON Files & Full-Qualified Translation Keys_. The elimination of nesting simplifies an i18n solutions in several ways.
   - Makes locale files more human-readable by flattening locale resources.
   - Reduces mistakes made by humans or automation by making files human-readable and using less indentation.
   - Facilitates Domain-Driven Design by forcing the developer to fully-qualify namespaces by domain.
   - Minimizes syntactical complexity of implementations by reducing checks for object vs string values.
   - Helps Translator plugins target Translation Values with a single key.
1. _Taxon Conventions_. It is likely necessary right out of the gates to provide Taxon Substitution so that Translation Values, themselves, can reuse text housed within the locale file, itself, to interpolate specific taxa. In fact, the 'details' object's `scope` property can be used in tandem to provide the appropriate taxon key, based upon user-driven data.
1. _Automation_. Oftentimes it may be necessary to write a small CLI, using something like [Inquirer](https://www.npmjs.com/package/@inquirer/prompts), to aid developers in committing entire translation-sets across all targeted locales with validation and rollback behavior.
1. _Product Owners_. If Product Owners are responsible for providing all translation values, a team may benefit from hosting a small backoffice app on your Staging environment with a minimal UI for CRUDing locale JSON files connected to a small Node API to handle read/write of the files, themselves.

#### TranslationService
##### `@core/utilities/l10n/translation.service.ts`
```typescript
const { LANG } = env;
const { location, navigator } = window;
const { searchParams } = new URL(location);
const { language: l, lang = LANG || searchParams.get('lang') || l } = navigator;
const DIALECT_EN_US = 'en-US';
const DIALECT_ES_MX = 'es-MX';
const LANGUAGE_EN = 'en';
const LANGUAGE_ES = 'es';
const DEFAULT_LANGUAGES = {
    [LANGUAGE_EN_US]: DIALECT_EN,
    [LANGUAGE_ES_MX]: DIALECT_ES,
};

export class Translator {
    dialect = LANGUAGE_EN_US;
    language = DEFAULT_LANGUAGES[this.dialect];
    languages = DEFAULT_LANGUAGES;
    locales = { };
    translators = new Set();
    locale = this.locales[this.language];
    
    constructor(options = {}) {
        const { dialects, locales, translators } = { ...this, ...options };
        
        this.dialects = dialects;
        this.locales = locales;
        this.translators = translators;
        this.change(lang);
        
        return this;
    }
    
    change = (dialect = DIALECT_EN_US) => {
        const { dialects, locales } = this;
        const { [dialect]: language } = { ...DEFAULT_LANGUAGES, ...dialects };
        const { [language]: locale } = locales;
        
        this.language = language;
        this.dialect = dialect;
        this.locale = locale;
        
        return dialect;
    };
    
    translate = (key, scope = {}) => {
        const { locale, translators, dialect, language } = this;
        const { [key]: value } = locale;
        return [ ...translators ].reduce( (value, fn) => fn.call({ dialect, language, key, locale, scope }, value), value );
    };
    
};
```

#### Basic Usage
##### `@app/children/**/*.component.ts`
```typescript
import { execute, change } from '@app/utilities/l10n';

export class UIElement extends HTMLElement {
    
    handleEvent(e: MouseEvent) {
        change('es');
        this.shadowRoot.innerHTML = this.render();
    }
    
    connectedCallback() {
        this.shadowRoot.addEventListener('click', this, true);
    }
    
    render() {
        const TITLE = translate('DOMAIN:SUBDOMAIN:TRANSLATION:KEY');
        return `<h1>${TITLE}</h1>`;
    }
    
};
```

#### Translator
##### `@core/core/utilities/l10n/translators.ts`
```typescript
import { utilities } from '@core/utilities';

export const translators = new Set()
    .add(translateInterpolation)
    ;
export {
    translateInterpolation,
};

function translateInterpolation(value: string): string {
    const { key, locale, scope } = this;
    const context = Object.assign({}, scope, { locale });
    return utilities.interpolate(value)(context);
}
```

##### `@app/core/utilities/l10n/translators.ts`
```typescript
import { marked } from 'marked';
import { utilities } from '@core/utilities';
import { translators as core } from '@core/utilities/l10n';
import { default as ltt_en } from '@app/children/**/TAC.en.md' with { type: 'text' };
import { default as ltt_es } from '@app/children/**/TAC.es.md' with { type: 'text' };

const translateMarkdown = {
    dialect: '',
    cache: '',
    
    call({ dialect, key, locale, scope }, value: string): string {
        if (key !== 'USER:SETTINGS:PRIVACY:TAC') return value;
        if (dialect === this.dialect) return this.cache;  // return cached result if dialect is the same as last run
        const context = Object.assign({}, scope, { locale });
        const ltts = { en: ltt_en, es: ltt_es };
        const text = utilities.interpolate(value)(ltts);  // interpolate the proper markdown content into translation value
        const interpolated = utilities.interpolate(text)(context);  // interpolate context variables into markdown
        
        this.dialect = dialect;
        this.cache = interpolated;
        
        return marked.parse(interpolated);  // parse as HTML
    }
    
};

export const translators = new Set([ ...core ])
    .add(translateMarkdown)
    ;
export {
    translateMarkdown,
};
```

### Known Uses
Perhaps to best example of usage for The Translator Pattern can be seen in use by the [AsXS](https://cscarlson.github.io/asxs) SDK. If you inspect the source code you'll fine that one of the Frameless&trade; components, the "copyright.rmd.html" partial leverages an instance of The Translator Pattern. Other examples may be found in the wild for various companies, however.

No known related patterns exist at the time of this writing. However, this pattern leverages The Chain of Responsibility Pattern under the hood to accomplish piping translations through the Translator plugins middleware.
