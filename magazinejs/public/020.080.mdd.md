
Mock-Driven Development (MDD)
================================================================

## Myth 1
_You need a framework like Axios for HTTP Requests_.

## Myth 2
_You need a framework like Storybook for Playgrounds and Component Catalogs_.

## Myth 3
_You need a framework like Sentry or LogRocket for logging & telemetry_.

## Reason
All of these disrupt Mock Driven Development practices if coupling to them directly.

## What Is MDD?
MDD is the strategy of employing heavy use of mocks in all approaches of software development from the outset. The benefits are numerous and solve many of the problems that plague us on the frontend.

### Benefits of MDD
- Development Velocity: tighter feedback loops while building, running, and writing tests.
  - Cross-network calls: fetching data is instantaneous.
  - Control over data: when you have control over what the data looks like, you require fewer feedback loops to write code.
  - Tests: Unit and Functional Tests reuse the same data as your catalog/playground (see below).
- Fault Tolerance: running the UI against mocks before real data discovers hidden anomalies in your code you were unaware of.
  - API outages: backend teams can work in parallel, making breaking changes, without impacting the UI until you go live (locally).
  - Network outages: ISP outages, VPNs, and other problems do not lower development velocity.
  - Airplane Mode: developers traveling on airplane need not pull the company card out for onboard wifi service.
- Quality Assurance: QA Engineers are happier when they can spoof operations in different environments.
  - State Loaders: Console APIs can be built for different environments so QAEs can instantly jump to specific user flows & states.
- Component Catalogs/Playgrounds: developing in a playground makes your code more portable when running it live (locally).
  - Stakeholder Transparency: providing a Component Catalog to your Product or Design team leads to cross-team productivity. 
  - Debugging: debugging in a playground/catalog allows you to tweak mimic production issues easier.
- Testing: mock data can be reused across different tests and different types of tests that run faster.
  - Unit Tests: Unit Tests can use the same data from your catalog.
  - Functional Tests: Fn Tests run as fast as possible and so can replace many of your heavy, cumbersome Unit Tests.
  - Smoke Tests: Smoke Test all of your components from a single component catalog.
  - Flakiness: MDD narrows test flakiness down to those only related to differences in environment.
- Telemetry: easily implement flexible telemetry & logging across any environment.
  - XHR: interjecting on network operations reduces side effects collected in telemetry services.
  - Logging: by always import and swapping out the `console` object per env eliminates the need to upgrade logging subscriptions.
- CI/CD: faster CI systems can be build without infrastructure to setup tokens, containers, and other configurations.
  - Pull Request Gating: quickly reject PRs that don't pass a test.
- Demo-Safe Instances: Developers, Product Owners and Sales personnel can safely demo instances that do not humiliate the company.

As you can see, we overcome _many_ of the common plagues we as frontend developers experience -- portability, entropy, fault tolerance, ... the list is massive. Here's roughly how to employ this approach.


## A Word On The Abstract Factory Pattern
As you may notice in the next article, we are beginning to [loosely] implement **The Abstract Factory Pattern**. The Abstract Factory Pattern does what _The Factory Method Pattern_ does but on a larger scale. That is, it uses a "Factory" that provides _Families_ of modules as opposed to a distinctly different _individual_ module. While each _file_ loosely implements The Factory Method Pattern, as a whole, our app receives an entire "family" of modules by simply importing each one as necessary.

## Synopsis
We demonstrated why you generally don't want to spring for a framework that does everything `fetch` already can. It is tougher to normalize against a library that you do not control, know little to nothing about its internal workings, or may change at the whim of personnel not on your team. Wrapping your telemetry service and importing your `console` at the beginning of a project can save you from expensive subscription price increases when you cannot afford to refactor unwrapped invocations all over your codebase. Once you have your 'Environment Factory' setup, you can easily build out something like a _HashRouter_ that uses CSS `:target` for showing and hiding elements in a _very_ plain Playground that speeds up development and acts as a Component Catalog which stakeholders can view for feedback, testing, technical User Story details, and refinement. Ultimately, the more you employ these techniques across a Mock Driven Development approach, the more benefits you'll stack up from the list in this article's opening.
