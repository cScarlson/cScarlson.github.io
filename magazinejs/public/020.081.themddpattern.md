
The MDD Pattern
================

### Intent
Define a development approach that employs heavy use of mocks throughout the software development lifecycle to improve development velocity, fault tolerance, and testing reliability.

### Also Known As
- The Mock-Driven Development Pattern
- Mock-First Development
- Mock-Driven Architecture

### Motivation
Traditional development approaches often couple components directly to external dependencies, creating bottlenecks in development, testing, and deployment. Mock-Driven Development addresses these issues by providing a systematic approach to mocking that enables parallel development, testing without infrastructure dependencies, and consistent environments across all stages of development.

### Applicability
Use the mdd pattern when
- you need to develop frontend components before backend APIs are complete.
- you want to eliminate network dependencies during development.
- you need to test components in isolation from external services.
- you want to create component catalogs that work without backend dependencies.
- you need to test error scenarios that are difficult to reproduce with real services.
- you want to improve development velocity by eliminating waiting times.

### Participants
- Mock Environment: Provides mock implementations of external dependencies.
- Environment Switcher: Allows switching between mock and real environments.
- Mock Data Provider: Supplies consistent mock data across development and testing.
- Service Adapter: Abstracts real services to maintain consistent interfaces.

### Collaborations
- The Mock Environment intercepts calls to external services and returns mock responses.
- The Environment Switcher determines which implementation (mock or real) to use based on configuration.
- Mock Data Providers ensure consistency between development, testing, and documentation examples.
- Service Adapters maintain interface compatibility between mock and real implementations.

### Consequences
The MDD Pattern
- Eliminates development bottlenecks caused by unavailable backend services.
- Enables parallel development of frontend and backend components.
- Provides consistent environments for development, testing, and documentation.
- Reduces test flakiness by removing network dependencies.
- Enables testing of error scenarios that are difficult to reproduce with real services.
- Improves development velocity by eliminating waiting times for external dependencies.

### Implementation
Implementation considerations include
- Creating environment-specific configuration files.
- Implementing service adapters that maintain consistent interfaces.
- Establishing conventions for mock data organization.
- Setting up build scripts that support different environments.
- Creating utilities for switching between environments.

### Sample Code

#### NPM `package.json` Scripts
Create scripts that act as an _Ignition Key_ for all relevant environments.

```json
{
    ...,
    "scripts": {
        "tst": "framework run --env tst",
        "tst:test": "test run --env tst",
        "mck": "framework run --env mck",
        "mck:test": "test run --env mck",
        "lcl": "framework run --env lcl",
        "lcl:test": "test run --env lcl",
        "dev": "framework run --env dev",
        "dev:test": "test run --env dev",
        "stg": "framework run --env stg",
        "stg:test": "test run --env stg",
        "prd": "framework run --env prd",
        "prd:test": "test run --env prd",
    },
    ...
}
```

#### Environments
Use an environment `type` driven from the CLI.

##### `@environment/index.ts`
```typescript
import { environment as mck } from './env.mck';
import { environment as dev } from './env.dev';
import { environment as stg } from './env.stg';
import { environment as prd } from './env.prd';

const { env } = process;
const { type } = env;

export const { [type]: env } = {
    'tst': mck,  // use mock for test
    'mck': mck,
    'lcl': dev,  // use dev for local
    'dev': dev,
    'stg': stg,
    'prd': prd,
    '':          prd,  // use fallback for missing env type
    'null':      prd,  // use fallback for missing env type
    'undefined': prd,  // use fallback for missing env type
};
```

#### Normalize Against `fetch` and `console`
Ditch Axios and other tools that `fetch` already resolves. Beside, security vulnerabilities have already been discovered in Axios and other 3rd-party packages.

##### Always Import `fetch`
###### `@core/utilities/http/fetch.ts`
```typescript
import { env } from '@environment';
import { default as json } from 'raw!@mocks/http/my-reponse.json';

const { TYPE } = env;
const { fetch: native } = window;
let fetch = native;

async mock(input: URL | RequestInfo, init?: RequestInit): Promise<Response> {
    return new Response(JSON.stringify(json), { status: 200 });
}

if ({ 'mck': true, 'test': true }[ TYPE ]) fetch = mock;

export { fetch };
```

##### Swap Your `Console` Object
###### `@app/utilities/telemetry/index.ts`
```typescript
import { env } from '@environment';
import { console as mck } from './env.mck';
import { console as dev } from './env.dev';
import { console as stg } from './env.stg';
import { console as prd } from './env.prd';

const { TYPE } = env;

export const { [TYPE]: console } = {
    'tst': mck,  // use mock for test
    'mck': mck,
    'lcl': dev,  // use dev for local
    'dev': dev,
    'stg': stg,
    'prd': prd,
    '':          prd,  // use fallback for missing env type
    'null':      prd,  // use fallback for missing env type
    'undefined': prd,  // use fallback for missing env type
};
```

The benefit of always importing your console this way is that you can control what specifically gets output in each environment and divert or mute different data and telemetry. It is recommended to also use this with a convention in your invocation signatures so that the first parameter of is always `@{env}`, where `{env}` is either `mck`, `dev`, etc. Muting a log is as simple as an Early Return and diverting your telemetry to a service is just as easy. Another nice thing about this approach is that your entire codebase simply calls `console.log` without caring what the actual logging strategy behind it is; so you can even swap out your production telemetry service with a different provider if your subscription price increases.

###### `@app/utilities/telementry/console.dev.ts`
```typescript
import { Console } from '@core/utilities/telementry/console';
export const console = new (class DevConsole extends Console {
    log(id: string, ...splat: any[]): void {
        if (id === 'too') return;
        if (id === 'much') return;
        if (id === 'console') return;
        if (id === 'noise') return;
        super.log(id, ...splat);
    }
})();
```

###### `@app/utilities/telementry/console.prd.ts`
```typescript
import { Console } from '@core/utilities/telementry/console';
export const console = new (class ProdConsole extends Console {
    log(id: string, ...splat: any[]): void {
        if (id === 'token') return;  // no-no
        super.log(id, ...splat);
    }
})();
```

#### Mock Data Access
There are basically 3 ways to implement the actual mock data.
- hardcode various `new Response`'s (not recommended): mock imports are carried into bundle.
- serve a static directory: you must use folder/file naming conventions and append "`.json`" to fetch URLs.
- Web Worker (recommended):
  - the worker only gets attached in `mck` mode.
  - intercepts `fetch` events.
  - imports mock data that never gets bundled.
  - can use conventions for if the mock is a `Response`, `Request` (makes request), or a `function` (that returns a `Response`)

## Known Uses
- Frontend development teams working with incomplete backend APIs.
- Testing environments that require consistent data.
- Component documentation and catalog systems.
- Development of offline-first applications.
- Performance testing without network dependencies.

## Related Patterns
- The Adapter Pattern: Used to maintain consistent interfaces between mock and real implementations.
- The Abstract Factory Pattern: Used to create appropriate service instances based on environment.
- The Strategy Pattern: Used to switch between different implementations based on configuration.
- The State Pattern: Used to switch between different modules based on configuration.
