
The Environment Flag Pattern
================================================================

## Intent
Define a system for controlling feature availability and behavior across different environments without requiring a full-featured feature flag framework or external service.

## Also Known As
- The ENV Flag Pattern
- Environment-Based Feature Control
- Simple Feature Toggling

## Motivation
Feature flag frameworks often introduce unnecessary complexity and dependencies for applications that primarily need to control features between development, staging, and production environments. The Environment Flag Pattern provides a lightweight approach to feature control that leverages environment configuration rather than runtime flag management.

## Applicability
Use the environment flag pattern when
- you need to control features between different deployment environments.
- you want to enable features for testing before production release.
- you need to disable debug or development features in production.
- you want to implement simple A/B testing without external services.
- you need to control features based on environment type rather than user segments.

## Structure
```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   Environment   │───▸│  Feature Flag   │───▸│   Component     │
│   Configuration │    │   Evaluator     │    │   Behavior      │
└─────────────────┘    └─────────────────┘    └─────────────────┘
         │                      │                      │
         ▼                      ▼                      ▼
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│   Environment   │    │   Probability   │    │   Conditional   │
│   Variables     │    │   Calculator    │    │   Rendering     │
└─────────────────┘    └─────────────────┘    └─────────────────┘
```

## Participants
- Environment Configuration: Defines the current environment and associated feature settings.
- Feature Flag Evaluator: Determines whether a feature should be enabled based on environment.
- Probability Calculator: Handles A/B testing by calculating feature probability.
- Component Behavior: The actual component or code that changes based on flag status.

## Collaborations
- The Environment Configuration provides the current environment type.
- The Feature Flag Evaluator checks configuration to determine feature status.
- The Probability Calculator handles random assignment for A/B testing scenarios.
- Component Behavior changes based on the evaluation results.

## Consequences
The Environment Flag Pattern
- Eliminates the need for external feature flag services.
- Provides simple environment-based feature control.
- Enables A/B testing with minimal overhead.
- Creates clear separation between environments.
- Reduces complexity compared to full-featured flag systems.
- Allows for feature rollout without code deployments.

## Implementation
Implementation considerations
- Create environment-specific configuration files.
- Implement a simple flag evaluation function.
- Consider probability calculations for A/B testing.
- Use URL parameters for auxiliary routes when needed.
- Maintain clear documentation of available flags.

## Sample Code
### Method Level Example
```typescript
import { LitElement, customElement, html, nothing } from 'lit';
import { env } from '@environment';

const { TYPE } = env;

export customElement('sdk-component') class MyElement extends LitElement {
    
    render() {
        if (TYPE === 'prd') return nothing;
        return html`
            <div class="non prod feature">
                ...
            </div>
        `;
    }
    
};
```

### Module Level Example
```typescript
import { LitElement, customElement, html, nothing } from 'lit';
import { env } from '@environment';

const { TYPE } = env;

class MyReleaseCandidateElement extends LitElement {
    
    render() {
        return html`
            <div class="non prod feature">
                ...
            </div>
        `;
    }
    
}

class MyNoOpElement extends LitElement {
    
    render() {
        return nothing;
    }
    
}

export const MyElement = customElement('sdk-component') {
    'mck': MyReleaseCandidateElement,
    'lcl': MyReleaseCandidateElement,
    'dev': MyReleaseCandidateElement,
    'stg': MyNoOpElement,
    'prd': MyNoOpElement,
    'undefined': MyNoOpElement,
    '': MyNoOpElement,
}[ TYPE ];
```

This works just fine for temporary Release Candidates and avoiding development bottlenecks. However, what if we want something a bit more powerful for something like A|B / Multivariate Testing? For that, whether or not we need a backend to drive this might depend on what our exact goals are. If all we need is a simple 50/50 split (or some other ratio), we can still setup a system in our frontend that processes a simple probability and maps it to a string that is easier to map to different features. That might looks something like the following.

### Probability System: `@app/feature/config.ts`
```typescript
const group = calculate();

function calculate() {  // returns 1, 2 or 3
    ...
}

export const { [group]: funnel } = {
    '1': 'campaign1',
    '2': 'campaign2',
    '3': 'campaign3',
};
```

## Known Uses
- Frontend applications with multiple deployment environments.
- Projects requiring simple feature toggling without external dependencies.
- Applications implementing gradual feature rollouts.
- Systems needing environment-specific behavior differences.
- Projects implementing simple A/B testing without third-party services.

## Related Patterns
- Feature Toggle Pattern: More complex version that supports runtime flag changes.
- Environment Pattern: Provides the foundation for environment-specific behavior.
- Strategy Pattern: Can be used to implement different behaviors based on flag status.
- Abstract Factory Pattern: Can be used to create environment-specific implementations.
