
Bootstrap is a library. If you're calling it framework you're using it wrong.
================
Import the SCSS and mix it in with things you've name yourself. Don't litter the markup with twenty strongly coupled names.

##### HTML
```html
<app class="todo app view">
    <header class="view header">
        <logo class="header logo"></logo>
        <h1 class="header title">Some Title</h1>
        <subheader class="header toolbar">
            <button class="toolbar button menu">
                <icon class="menu icon"></icon>
            </button>
            ...
        </subheader>
    </header>
    <main class="view body">
        ...
    </main>
    <footer class="view footer">
        ...
    </footer>
</app>
```

##### SCSS
```scss
@use "~bootstrap/dist/scss/buttons";
@use "~bootstrap-icons/font/bootstrap-icons.scss";

.todo.app.view {
    [...styles];
    
    .view.header {
        .header.logo { ... }
        .header.title { ... }
        .header.toolbar {
            [...styles];
            
            .toolbar.button {  // <-- extend button classes
                @extend .btn;
                @extend .btn-sm;
                [...additional-styles];
            }
            
            .toolbar.button.menu {
                [...styles];
                
                .menu.icon {  // <-- extend icon classes
                    @extend .bi;
                    @extend .bi-list;
                    [...additional-styles];
                }
                
            }
            
        }
    }
    
    .view.body { ... }
    .view.footer { ... }
    
}
```

## Synopsis
The main advantage of this is that your Markup looks how you want it to look without potentially dozens of classes in an elements className. Moreover, you get to take advantage of the HTML/CSS pattern mentioned in this repo (see "CSS Modules: Why you don't need them..."). On top of that, there is a FEFO ("Fail Early, Fail Often") rule that you benefit from as well. If, say, a Bootstrap deprecates the `.btn` class in favor of `.button` then you actually get an error message at development time -- even likely if you are not working in the part of the app or site using that class -- instead of just going to production to find out that the user saw a "button" that looked nothing like an actual button. This is not just better for the User Experience but also seems like a much better Developer Experience.
