
Environment Flags
================
At times we may jump through expensive hoops to implement _Feature Flags_ when all we need is to put something behind an _Environment Flag_. Rather than springing for a framework and a cloud subscription for the tool, it is likely that we can just leverage something like the following...

```javascript
import { env } from '@env';

const { TYPE } = env;
let service = new NoOpService();  // use noop for production

if ({ 'mck': true, 'dev': true, 'stg': true }[ TYPE ]) service = new Service();

export { service };
```

## Synopsis
Here we have a mechanism that uses an Inline Map to check if the environment type is either mock, dev, or staging. If so, then we swap out the service that gets exported with the real version. The real version is what we are currently developing but not ready to release yet, so we provide the No-Op version for production. This ends up looking a lot like _The Factory Pattern_, although, we can even use barrel files to implement something more akin to _The Abstract Factory Pattern_ so as to export entire _families_ of modules based upon the environment. Usually, however, the need is relatively short-lived or few & far between so a quick little swap like this does all you need. This can even compliment Mock Driven Development where you have an abstract class that is extended by a service module for each environment while the MDD service only provides No-Op methods, modifies the URL, or straight up returns hardcoded mock data.
