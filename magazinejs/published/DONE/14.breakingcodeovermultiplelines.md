
Why Your Professor Broke Their Code Out The Way They Did
================
Oftentimes we see code that leverages both virtical and horizontal space appropriately. This benefits the team & project by making it so, hopefully, a developer can open a file and immediately get a snapshot of what all the code is doing without having to scroll. There has always been a rule not to make files too large, to break things out into separate files, even if most of the code is similar. Sometimes it is difficult if all of the code exists within a single module. But even that can oftentimes be remedied by extracting various faculties into a separate `utilities` module.

An MIT study concluded that there are two aspects of code which most impact readability: (1) File-Length and (2) Indentation [<sup>citation needed</sup>]. These two aspects are rather correlated with one another. That is: the more indentation in the code, the longer the file-length. Makes sense. Getting deeper, the reasoning may likely be related to what some call an ["Event Boundary"](https://news.nd.edu/news/walking-through-doorways-causes-forgetting-new-research-shows/#:~:text=%E2%80%9CEntering%20or%20exiting%20through%20a,because%20it%20has%20been%20compartmentalized.%E2%80%9D). Basically, you remember there is something you need to do, go into the other room to do it, and all of a sudden just forget what your deliberate task was. It is attributed to passing through doorways/ports and other sorts of event boundaries. It is likely that _scrolling_ would have the same effect. So, having as much in the same IDE _viewport_ as possible without sacrificing readability is probably leads to higher comprehension of the code as you are not entering and exiting different viewports.

Beside all that: if there is horizantal space, why not take advantage of it anyway. Even something that runs _slightly_ off the right-hand side is remedied rather easily with a quick two-fingered flick on the touchpad. Running too far over horizontally is equally bad, however. This is when you actually _do_ want to break things out over multiple lines -- not just because. But when it is done out of a rule of thumb that has no consideration for horizontal space usage, we make the code less readable. After all, are we assuming everyone is going to be reading the code on their phones? One strong argument I have heard for this is that "I like to use the IDE's split-pane feature". Ok... well, then split it horizontally instead of making your teammates crazy and making the project suffer.

A lot of times I will come across some code that looks like this...
```html
<input 
    id="search" 
    class="form-control typeahead typeahead-small" 
    type="text" 
    placeholder="{placeholders.search}" 
    />
```
This is not _the worst_ if that is all it is. But we can see that it _does_ fit and read fine on just one line...
```html
<input id="search" class="form-control typeahead typeahead-small" type="text" placeholder="{placeholders.search}" />
```

It gets 10X more unwieldy when you have elements wrapping other indefinitely-nested elements, all of whom's attributes are broken over multiple lines for no sufficient reason.

Imagine if every book you bought used the same approach as example A. Your friends would think you are so smart because you could read a 1000-page book in the time it takes to read a 250-page book! Yeah... that's not smart, that's pretension. Those are roughly the numbers that I see in code all the time: I will condense a 1000-line file down to about only 250-lines. Remember: **you write code for _humans_, not _machines_**. Otherwise, our keyboards would have only two buttons -- `1` and `0`.

## What is _really_ going on here...
It is probably conspicious that the only time you really see this terrible practice is from those devs arriving in the workforce from _Code Bootcamps_. Seemed odd to me when I realized that. Then I put myself in their shoes and spoke with a few bootcamp grads to realize: _this is actually how they learned it_.

If you are one of these developers writing code like example A -- STOP! Did you ever take a moment to ask _why_ your professor was writing their code this way? Your professor had to make a relatively tough decision when designing their curriculum: put their code and their notes on separate slides -- OR -- display them within the same slide. It's an obvious solution: you go with putting them on the same slide for obvious reasons.

I have seem insurmountable video tutorials do the same. What's most shocking to me is that these professors & tutorial authors never seem to actually mention it. I have seen _some_ state that "by the way, the reason my code is broken out all crazy like this is because I had to make room for my notes on the same slide". In fact, the trade-off the profesor/author is actually making is likely due to an awareness of _Event Boundaries_. They likely understand that _if you have to change pages or scroll an entire page to look between both the notes and the code, you will likely not gather all the information without scrolling back & forth 20 different times_.

## Synopsis
Quit breaking EVERYTHING out over multiple lines, just because. It is solely because your professor had to display both the code and the notes in the same slide and they neglected to mention it, assuming you would know better. So, now, there are thousands -- tens of thousands -- of poor developers thinking this is the right way to write all of their code. There is nothing wrong with this practice when you've run out of horizontal space, but it is actually Bad Practice to do it by default.
