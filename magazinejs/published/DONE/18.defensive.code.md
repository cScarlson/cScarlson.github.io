
Writing Defensive Code
================
It is rather commonplace for a software shop to enforce something of a styleguide. It is also rather commonplace to find a shop not using a styleguide at all. On one end of the spectrum you have shops that don't have a styleguide and on the other end of the spectrum a shop that strongly enforces a very strict styleguide. This article offers an alternative.

## What _Exactly_ Is a Styleguide, anyway?
A styleguide is ... _something that **guides** style_; _style_ being the highly nuanced manner in which one writes code. This _style_ has everything to do with a developer's path of learning and, thus, their _productivity when writing code_. Therefore, this is not something we want to tamper with too much.

### The Problem With Styleguides
With the definition above, you may notice that styleguides are rarely something akin to a "guide". Rather, they tend to be merely documentation of highly strict _dicta_, not a _guide_. The documentation for these dicta, the "styleguide", end up looking like a document that targets _extremely low-level_ "do's" and "dont's" -- essentially _micromanaging_ developers' styles enough to make your local USPS office workers vomit.

## A Better Alternative
What if we could (a) _be more comfortable (or at least less OCD) about our peers' **style**_ and (b) _provide them with a better intuition as their particular style's **guide**_? Then we wouldn't really have to show them every possible do/don't and continually add new ones to our documentation. The better alternative? **intuition for writing defensive code**. You write code for humans, not machines. The way you shoud be thinking about the code as you write it should always be: "how do I communicate with the future developer what the code does so they can more easily maintain it?" -- because that "future developer" might just be you later on. The goal should be to write it in such a way that another hand only has to 'add a similar line' to what is already there. That might resemble putting things in a lookup Map or Set, or even just adding another clause for an Execution Guard or Early Return.

## A Controversial Word on Controversy
As an experienced developer you ought to be comfortable with confrontation. That is the basic priciple of a Code Review. And confrontation can cause controversy. You look at each others code and confront the parts which may have potential problems. At times these "problems" may boil down to small deviations in preferences that have arbitrary tradeoffs. That is all good and well as most comments in a Code Review should be provided and received on a _take it or leave it_ basis. Those that do not fall into that category are not unlike security concerns, unnecessary deviations from architecture, etc. Those that do fall in that category are likely to stir controversy because either the author or reviewer was inexperienced in confronting concerns with constructive criticism.

### The Driving Force of Guiding Style Should be Intuition Sourced from Principles
A proper styleguide should not be restrictive but empowering one's intuition to generally handle any given coding situation or dilemma. Needless to say, less experienced developers may need more handholding but more experienced devs should generally have the respect of handling themselves, not being micromanaged by a hyperspecific ever-growing ruleset that tries to identify every possible situation.

### Multiline `if` Statements Should Be the Exception, Not the Rule.
I know everyone and their Grandmother learned to "always use braces for `if`/`else` clauses". This is the opposite of what you should do.

#### Reasoning
If an `if` clause is complicated enough to require multiple lines, the behavior should be tucked away into a nicely named function that indicates the operation. Also, indentation & file-length are the top 2 leading criteria that impact readability. Keeping your code flatter makes it more readable. **Anytime you write code in multiline block braces you are inviting corruption to that code with every hand that touches it -- including yourself 6-months down the road**.

## Early Returns / Execution Guards
We've all seen heinous code like this...
```javascript
function modifySomething(type, id, name) {
    if (type) {
        if (id === 'a') {
            if (name !== undefined) {
                // probably
                // lots
                // of
                // redundant
                // code
                // across
                // a
                // b
                // c
                // ... and
                // z
            } else {
                name = 'default';
            }
            
            return 'x';
        } else if (id === 'b') {
            // see clause a ...
            return 'y';
        } else if (id === 'c') {
            // see clause a
            return 'z';
        } else {  // assume id === 'z'
            return null;
        }
    } else {
        return null;
    }
}
```
Dear lord... How is anyone supposed to know what might be going wrong when they debug this unless they carefully spend [too much] time setting a breakpoint on just about every single line and rerunning it a thousand times. How can we have any reliable clue that we are _always_ returning something we expect in this code and not ever `undefined` (because there is no `return` other than those within if/else blocks)? Moreover, how do we know that `if (name !== undefined)` is not misleading? Is it just if it's _falsey_? Or is it ok if `name` is an empty string (`''`) or `null`? "This ...", as my Mentor in 2013 might say, "... is dog sh!t. I mean, PURE, dog sh!t". And he would be correct.

### First Thing...
What in the world was going through their head when they wrote a huuuuuuuuge `if` block and then stubbed in a single little `return null` in the `else` clause? So the first thing we should hope for is some boolean inversion at the very least...

```javascript
function modifySomething(type, id, name) {
    if (!type) {
        ...
    }
    
    return null;
}
```
Now we can at least assume _somethng_ will _always_ be returned (other than `undefined`/void).

### Solution
```javascript
function modifySomething(type='default', id, name='default') {
    if (!type) return null;  // #execution-guard
    if (!id) return null;    // #execution-guard
    if (!name) return null;  // #execution-guard
    const { ... } = DEFAULTS;
    // ... more setup for more variables
    
    if (id === 'a') return modifySomethingForId(id, name);  // #early-return
    if (id === 'b') return modifySomethingForId(id, name);  // #early-return
    if (id === 'c') return modifySomethingForId(id, name);  // #early-return
    return null;
}

function modifySomethingForId(id, name) {
    if (!id) return null;    // #execution-guard
    if (!name) return null;  // #execution-guard
    // otherwise
    // lots
    // of
    // redundant
    // code
    // across
    // a
    // b
    // c
    // ... and
    // z
    return null;
}
```
What a world of improvement. Notice a couple of things about the solution...

- the code is flatter (thus more readable).
- the code is more SRP (see SOLID).
- the code is less redundant.
- it is easier to tell what scenarios we _do_ (and don't) target/capture.
- we know, _for sure_, we are _always_ going to return something (`null` by default).
- the code will _always_ either (a) hit an Execution Guard or (b) run in a _HappyPath_ manner (the way we expect).
- all we had to do is invert some conditions, add some Execution Guards & Early Returns, and wrap redundant code in a function.
- maintaining this code just means adding more Execution Guards or Early Returns.

## Named Operations, Anonymous Functions, Inline Arrow-Functions/Lambdas and Multiline Inline Declarations
First off, **always name your ops!**. Anonymous functions _do_ have their place, but that place is not _everywhere_. The issue with anonymous functions is that (a) they can lead to a rather useless debugging experience and (b) they actually make the JavaScript compiler (yes, compiler) have to work harder [<sup>according to Kyle Simpson</sup>]. The other issue -- a **large issue** in my opinion -- is that you cannot "name your ops" with anonymous functions. Naming your operations has several benefits

- easier debugging
- clarity of intention
- identification of abstraction

The latter is the big one. If everything is an anonymous function then it becomes _much_ more difficult for another hand to come by and extrapolate behavior that can be factored into an abstract function, which behavior can be wrapped by another function, and when an entirely new function should be created for similar but different behavior. Take this code for example...

### Poor Maintainability
```jsx
import React, { useEffect } from 'react';

const DEFAULT_PROPS = {
    ...
};

function Component(props) {
    const { x, y, z } = { ...DEFAULT_PROPS, ...props };
    
    useEffect(() => {  // operation A
        // subroutine 0
        // subroutine 1
        // subroutine 2
        // subroutine 3
        // subroutine 4
        // subroutine 5
    });
    
    useEffect(() => {  // operation B
        // subroutine 0
        // subroutine 1
        // subroutine 2
    }, []);
    
    useEffect(() => {  // operation C
        // subroutine 3
        // subroutine 4
        // subroutine 5
    }, [ x ]);
    
    useEffect(() => {  // operation D
        // subroutine 0
        // subroutine 1
        // subroutine 2
    }, [ x, y, z ]);
    
    return (<>...</>)
}
```
As you can see from the comment labelling, there are redundancies in the code. If these were named we would have an even clearer idea of where the commonalities exist. Not to mention: these are all _inline declarations_ that discourage the next person from pulling them out and naming the ops themselves. Moreover, the next person will likely have little to no understanding of the original intention and be encouraged to just modify each operation in place to disturb the code as little as possible -- leading to more corruption of the code. That is, _this code invites corruption_. And this is before we even get into returning any cleanup functions that might look similar.

### A Better Way
```jsx
import React, { useEffect } from 'react';

const DEFAULT_PROPS = {
    ...
};

function Component(props) {
    const { x, y, z } = { ...DEFAULT_PROPS, ...props };
    
    function mount() {  // operation A
        // subroutine 0
        // subroutine 1
        // subroutine 2
        // subroutine 3
        // subroutine 4
        // subroutine 5
    }
    
    function initialize() {  // operation B
        // subroutine 0
        // subroutine 1
        // subroutine 2
    }
    
    function observeX() {  // operation C
        // subroutine 3
        // subroutine 4
        // subroutine 5
    }
    
    function observeXYZ() {  // operation D
        // subroutine 0
        // subroutine 1
        // subroutine 2
    }
    
    useEffect(mount);
    useEffect(initialize, []);
    useEffect(observeX, [ x ]);
    useEffect(observeXYZ, [ x, y, z ]);
    
    return (<>...</>)
}
```
Looking much better already. In fact, we may have actually shortened the code. And this might not be perfect in its current state, but it is a lot better than before because it is (a) _more improveable_ and (b) not inviting corruption into itself. Now let's see how the next hand might come by to improve it through refactoring the operations...

### After Refactor
```jsx
import React, { useEffect } from 'react';

const DEFAULT_PROPS = {
    ...
};

function Component(props) {
    const { x, y, z } = { ...DEFAULT_PROPS, ...props };
    
    function subroutine012() {
        // subroutine 0
        // subroutine 1
        // subroutine 2
    }
    
    function subroutine345() {
        // subroutine 3
        // subroutine 4
        // subroutine 5
    }
    
    function mount() {  // operation A
        subroutine012();
        subroutine345();
    }
    
    function initialize() {  // operation B
        subroutine012();
    }
    
    function observeX() {  // operation C
        subroutine345();
    }
    
    function observeXYZ() {  // operation D
        subroutine012();
    }
    
    useEffect(mount);
    useEffect(initialize, []);
    useEffect(observeX, [ x ]);
    useEffect(observeXYZ, [ x, y, z ]);
    
    return (<>...</>)
}
```
Ok, this code _might_ be _slightly_ longer (because of braces) but it is pretty negligible. Nonetheless, we get a massive benefit for the tradeoff -- our code is significantly _more improvable_ than it was before.

### (Re)Coupled
```jsx
import React, { useEffect } from 'react';
import { utilities } from '@core/utilities';  // <-- factored out into common utilities object

const DEFAULT_PROPS = {
    ...
};

function Component(props) {
    const { x, y, z } = { ...DEFAULT_PROPS, ...props };
    
    function mount() {  // operation A
        utilities.subroutine012();
        utilities.subroutine345();
    }
    
    useEffect(mount);
    useEffect(utilities.subroutine012, []);  // operation B
    useEffect(utilities.subroutine345, [ x ]);  // operation C
    useEffect(utilities.subroutine012, [ x, y, z ]);  // operation D
    
    return (<>...</>)
}
```
Again, a large improvement. Again, a reduction in code. Now that some little angel refactored this to provide clear and concise boundries of intention: not only do we have less code to figure out and stumble over -- we know that, when certain behavior needs to change across the board for a specific type of operation, we can change it in just _one_ spot. Likewise, we can add new behavior only where it actually belongs without struggling to figure out which places require it and which do not. In fact, now we might even be able to extract one or more of the subroutines out to put it into a `utilities` module that can be used across other components. In other words, we may have just _decoupled_ some of the code (from itself) -- suparating _abstraction from implementation_. Anytime we can decouple code it is always a _massive_ win. It not only frees one module but it also empowers other modules, making them all more SRP (see SOLID) by increasing their _Cohesion_. You might say, "but wait... I thought Cohesion _decreases_ as Coupling _increases_". True. What we actually did is _reorganize_ our coupling. We actually coupled this code with a new utilities module, so we indeed _increased_ our _Link Degree_ (_Fan-In/Fan-Out_). We decoupled the code from itself so as to accept the tradeoff of using it elsewhere. Just because we _abstracted_ it that does _not_ mean we _decreased_ its coupling. The benefit we gained by increasing the code's Fan-Out and increasing its Cohesion is that now we have more consistency & reliability across the entire system (as opposed to having to hunt & peck everywhere when a requirement changes).

#### Summary
Avoid using anonymous functions. Avoid using multiline inline declarations. When passing a function into another as a callback (or "predicate function") either use a single-line arrow-function/lambda or provide a named function. The only time you should not worry about "naming your ops" is when it is a simple single-line arrow-function. If it is multiline then the operation is complex enough (or will be soon enough) to warrant naming the operation to convey its intention. Not to mention: your code will become flatter and read nicer -- allowing the reader to approach the code in a Top-Down, Stepwise manner and even omitting low-level details that they might not be concerned with (at least before understanding the code at a lower resolution first). Lastly, when you actually define a function with the `function` keyword, it makes it a lot easier to immediately determine which declarations are actually _variables_ and which are actually _functions_. **Name your ops!!!**

### A Better Way

##### `**/some/other/domain/utilities.ts`  <-- you can name it the same thing under a different domain, bozo.
```typescript
import { env } from '@env';
import { utilities as core } from '@core/utilities';

const { API_URL } = env;
const convertToSSL = convertURL.bind(undefined, true);
const details = {
    modified: true,
};
let destroyed = false;

function canProvideFor(data, callback) {
    callback('parkingGarage' in data);
};

function execute(callback) {
    callback(API_URL);
};

function convertURL(secure = true, url) {
    if (!secure) return url.replace('https', 'http');
    return url.replace('http', 'https');
}

function initialize() {
    core.doSomeStuff();
}

function destroy() {
    destroyed = true;
}

export {
    canProvideFor,
    execute,
    convertToSSL,
    initialize,
    destroy,
};
```
Quite a bit more readable, I'd say.

### When Inline Exports Are Ok
Inline exports are actually great when you write something that is so basic it is almost impossible to be corrupted over time. Usually this is when you can logically only have a single simple export...

```javascript
export const constants = {
    SOME_CONSTANT: '...',
    ...
};
```
This is fine. An other example...
```javascript
export default {
    SOME_CONSTANT: '...',
    ...
};
```
... and ...
```javascript
import { default as constants } from './subsystem/constants';
```
Another example...
##### `./endpoints.ts`
```typescript
import { env } from '@env';

const { API_URL } = env;

export default {
    PROJECTS: `${API_URL}/projects`,
    PARTICIPANTS: `${API_URL}/participants`,
};
```
... and ...
##### `./index.ts`
```typescript
export { default as endpoints } from './endpoints';
...
```
##### `**/*.ts`
```typescript
import { endpoints } from '@some/domain/subsystem';

const { PROJECTS, PARTICIPANTS } = endpoints;

...
```
Just be mindful that if the file may likely eventually export more than just a single basic module then you are essentially inviting your code to be corrupted. I would advise: only use inline exports when you are certain of this; if you have any doubt, don't do it. Otherwise, all you are doing is inviting corruption into the code.

## Always Have an Ignition Button
What is an "ignition button"? Like a car, you need _some way_ to run the thing. Imagine if, with a car, there was no ignition-key operation to turn the motor over. Imagine that there were several ways: you could (a) rub some wires together underneath the steering wheel, (b) rub some wires together under the hood, (c) a push-start mechanism, or (b) a kick-start mechanism. Well, those are not bad options for secondary or tertiary options in case the main option fails, but without a main ignition mechanism you would likely have some people running it one way and others another just because they perhaps were not aware of other options. Moreover, you'd likely have people trying to rewire the electrical system just to make the one way work, or one of the other ways when the one fails, or completely new ways never anticipated. That's what happens with software when you don't provide a [main] ignition operation. Everyone's going to be running it a different way and someone's going to try to rewire the whole damn thing's internals just to get it to run. So there need be a main way and the main way better work for everybody, only giving them failover alternatives for rare chances.

### Ignition in `package.json`
```json

    "name": "ignition-example",
    "scripts": {
        "start": "framework start",
        "test": "runner --target='./**/*.spec.ts'",

        "deploy": "cli ./dist --dest='https://username:password@www.example.com'"
```

## Listy Objects: Growers vs Show-ers
First off, get your mind out of the dang gutter. What I mean by this is how & when to write a 'listy' object _inline_ vs _broken out_. First what's a "listy" object? _A listy object is just an array or object-literal/map that conveys some information for lookup or iteration_.

### Show-ers
```javascript
const types = [ 'user', 'admin', 'superadmin' ];
```
We write a list like this because (a) the available values are _limited_ and (b) there is _not_ a large amount of them. If this list was larger _or_ it would likely _grow_ as the app does (the list grows as more requirements arrive) then we would consider this list a "grower".

#### Another Example: _Inline Map_
```typescript
function handleCheckboxChange(e) {
    const { target } = e;
    const { value }: { value: 'true'|'false'|boolean|undefined|null|''|1|0 } = target;
    const checked = { 'true': true, 'false': false, 'undefined': false, 'null': false, '': false, '1': true, '0': false }[ value ];
    
    if (checked) ...;
    else ...;
}
```
This example is technically rather contrived but that's not the point. The point is: even though we have a rather robust list of possible values to map to (or keys), that list is _not ever growing any bigger_. We used an inline map because this code hypothetically had some sort of oneoff but the point is the same. Moreover, _Prettier_ would want to treat this (and every godforsaken object) as a _grower_ when it clearly is actually just a show-er. Of course we are bumping up against the line of a rational character-column limit, but this is easily remedied with a quick two-fingered flick on the touchpad and back. After all, is it really something you care about seeing once you have a quick glance at it? No, it is extremely trivial after that glance. That said, it is pushing the limits a bit so after adding a few more values to it you may want to pull it out, stick it somewhere nicely, and consider it a grower (by our definition above).

### Growers
```javascript
const handlers = [
    handleScenarioA,
    handleScenarioB,
    handleScenarioC,
    handleScenarioD,
    handleScenarioE,
    handleScenarioF,
    handleScenarioG,
    handleScenarioH,
    ...,
];
```
This list is a "grower" because we can assume that other usecases will arise before long and somebody will need to register another handler with our system. Another example of a grower may be a `routes` definition. Another: a `constants` object or an `endpoints` docket.


## Synopsis
Most "styleguides" are not style _guides_, they are style _mandate_ documents. More important than a styleguide is **writing Defensive Code**. The guiding principle behind writing Defensive Code should be one's intuition and that is what a style _guide_ really is: a document such as this that points to _a set of **principles**_ to help you build an intuition instead of a document that points to _a set of **low-level dictates for every currently known situation**_. This is important because (a) we cannot predict every possible situation that our OCD wants to _micromanage_, (b) the entire point is to avoid corruption in our code -- not micromanage experienced professionals, (c) nobody has their productivity slashed by having to conform to micromanagement "nits", and (d) Code Reviews are geared closer to how much the code _invites corruption_ instead whether or not you used a perfectly reasonable feature of the language that was "just unexceptable" in someone else's eyes (because of dogma, OCD, and micromanagement).
